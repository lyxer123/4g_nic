## 1.W5500 / esp_eth 初始化失败（串口红字）的调试，202604010818

1.1 什么现象

串口出现红色错误日志，例如：

- `w5500.mac: w5500_reset: reset timeout`
- `w5500.mac: emac_w5500_init: reset w5500 failed`
- `esp_eth: esp_eth_driver_install: init mac failed`
- `esp_eth: esp_eth_start: ethernet driver handle can't be null`
- 后续周期性：`w5500_send_command: send command timeout`、`emac_w5500_receive: issue RECV command failed`、`frame read from module failed`

含义概要：通过 RST 对 W5500 做硬件复位后，芯片未在预期时间内就绪，SPI 访问无有效应答，导致 MAC 初始化失败、`eth_handle` 为空；若仍有收包路径在跑，会持续报 SPI 命令/收帧超时。属**硬件或供电/SPI 物理层**问题，不是应用层 ping 配置问题。

同一块板、布线未改，仅更换供电环境（例如家里与办公室），也可能复现或消失：两块 12V/1A 适配器经 MP1584 转 5V/3.3V 后，**纹波、瞬态负载、插头压降** 不同，会使 3.3V 在尖峰电流下短时跌落，表现为偶发 `reset timeout` / `send command timeout`。

1.2 如何调试

1. 对照 `sdkconfig`：SPI（SCK/MOSI/MISO/CS）、INT、RST 与 PCB 是否一致；RST 是否与 4G 复位等 GPIO 冲突。
2. 万用表测 W5500 附近 **3.3V** 是否稳定（是否明显低于约 3.0V 或大幅抖动）。
3. **对照实验**：只换 12V 适配器或只换供电场所，观察日志是否跟随变化。
4. 在 3.3V 靠近 ESP/W5500 处加强 **去耦（陶瓷 + 可选 47～100µF）**；缩短 12V 输入线、压紧插头。
5. 软件侧：将 `CONFIG_BRIDGE_ETH_SPI_CLOCK_MHZ` **暂时降到 10MHz 或更低**，排除布线边际下的 SPI 误码。
6. 若原理图无 MCU 控制 RST，核对是否应关闭软件 RST GPIO 或接法是否反（低有效 RSTn）。

1.3 调试结果

当前结论（分析阶段）：红色日志对应 **W5500 未正确完成硬件复位或 SPI 无应答**，优先排查 **供电瞬态/适配器差异、RST 与 SPI 接线、本地去耦与 SPI 时钟**，而非 iot_bridge 业务逻辑。板级未改仅换供电环境即可复现差异时，**电源质量**为高度可疑因素。具体哪一项起主导需按 1.2 实测后在此补充（如适配器对调、加电容、降 SPI 后的串口对比）。

## 2.W5500 芯片 ID（0x04）与「mac select」日志的辨析，202604011430

2.1 什么现象

- 串口先有 **`mac select: MAC ec:da:3b:99:ee:04`**（或类似），随后出现红字：  
  **`W5500 version mismatched, expected 0x04, got 0x00`**、**`verify chip ID failed`**、**`esp_eth_driver_install: init mac failed`**、**`esp_eth_start: handle can't be null`**。
- 同时 **4G PPP** 仍可能 **`Modem Connect`**、**`NAPT enabled`**，容易误以为「网线 MAC 已读回、以太网也正常」。

2.2 如何调试 / 分析要点

1. **`mac select: MAC …`**：来自 **ESP32 本机**（eFuse / `esp_read_mac` 等），表示 **给 ETH_LAN 网卡配置的 MAC**，在 **创建 netif 阶段** 就会打印，**不是**通过 SPI 从 W5500 读出的芯片地址。
2. **`expected 0x04, got 0x00`**：驱动读 W5500 **版本/ID 寄存器**，真片应为 **0x04**；读到 **0x00** 多表示 **SPI 未接通**（MISO 总为 0、CS/线序错）、**芯片未上电/未出复位**，或 **时序/速率** 导致误读。
3. **`esp_eth_driver_install` 失败** ⇒ **`eth_handle` 为空**，**W5500 以太网未起来**；与 PPP 是否为 **两路独立外设**，互不为证。
4. 结合 **1** 中条目：供电、SPI 引脚、RST、CS、降 SPI 时钟等，**直至 ID 能稳定读出 0x04**。

2.3 调试结果

（记录用）**有影响**：只要出现 **`verify chip ID failed`**，当前启动下 **SPI 以太网不可用**；不能仅凭 **`mac select`** 判断 W5500 正常。后续可在排除硬件后在此填「读 ID 正常、Link Up」等结论。

## 3.ESP32-S3 上 Flash/PSRAM 与外设 SPI（及 ETH Menu 填写），202604011445

3.1 什么现象 / 背景问题

- 疑问：**SPI1 是否被内部 Flash/PSRAM 占用？是否只剩 SPI2、SPI3 给 W5500？**
- 疑问：**ETH Configuration 里 `SPI Host Number`、`PHY Address of SPI Ethernet Module` 填什么？**

3.2 如何调试 / 分析要点

1. **Flash /（若有）PSRAM**：走芯片 **MSPI 专用存储接口**，与 **XIP/cache** 绑定，**不是**应用里 `spi_master` 挂 W5500 的那套「通用 SPI 主机」。资料中的 SPI0/SPI1 与存储端口对应属 **SoC 内部命名**，勿与 Menu 里给外设选的 **Host 号**混为一谈。
2. **W5500 等外设**：应使用 **通用 SPI 主机**，在 ESP-IDF 中一般为 **`SPI2_HOST`、`SPI3_HOST`**（工程里 **`CONFIG_BRIDGE_ETH_SPI_HOST=1`** 通常对应 **`SPI2_HOST`**，具体以当前 IDF 版本 Menuconfig 说明为准）。**不要**与 Flash 走线混接。
3. **Menu — SPI Host Number**：选 **与 PCB 上 SCK/MOSI/MISO/CS 所接那一组 SPI 控制器一致**的值；填错会出现 **ID 读 0x00**、超时等，不能「随便试一个数」。
4. **Menu — PHY Address（SPI 以太网模块）**：对 **W5500** 无 MDIO PHY 地址含义，API 兼容字段，**单芯片场景一般固定填 `0`**（与 `CONFIG_BRIDGE_ETH_SPI_PHY_ADDR0=0` 一致）。多片/特殊硬件再按手册调整。
5. **结合 1、2**：硬件与 Host、PHY Addr 配置一致后，应用 **寄存器级** 验证（首选 **版本 ID = 0x04**）。

3.3 调试结果

（结论摘要）ESP32-S3 上 **存储用 MSPI**；外设 SPI **常用 SPI2/SPI3**。ETH Menu：**Host = 按布线选（常见为 2 号外设映到枚举 1 等）**；**W5500 PHY Address 一般用 0**。本条目为原理说明，具体板级以 **实测 ID 与 Link Up** 为准。

## 4.4G 转 WiFi 已通、4G 转 W5500（ETH_LAN）异常的调试，202604041200

4.0 适用版本与环境（重要）

本条目的 **现象复现、根因分析与对 `bridge_eth.c` 的补丁**，以当时工程为准；**换版本后不能假定仍适用**，需重新对照源码并做编译与联调。

| 项目 | 本仓库当时使用的版本 | 说明 |
|------|----------------------|------|
| **ESP-IDF** | **`v5.2.6`**（当前常用）、曾用 **`v5.1.6`** | **`v5.2.6`** 下 **W5500 可正常通讯**，见 **条目 8**；**v5.1.6** 为条目 4 补丁与现象记录时的基线；`dependencies.lock` 以实际 checkout 为准。 |
| **esp-iot-bridge** | **`1.0.2`**（组件 ID：`espressif/iot_bridge`） | `dependencies.lock` 与运行时 `bridge_common: esp-iot-bridge version: 1.0.2`。 |

- **其它 ESP-IDF 主版本**（例如 5.0、5.3、5.4、6.x）：工具链、lwIP、`esp_eth` 与 Kconfig 可能与 5.1.x 不同，**补丁行号、`#if CONFIG_BRIDGE_*` 分支、回调注册方式可能已变化**，需打开对应版本的 `managed_components/.../iot_bridge/src/bridge_eth.c` **逐段核对**后再改。
- **其它 esp-iot-bridge 版本**：上游可能已调整 `eth_netif_dhcp_status_change_cb` / `dns_change_cb` 逻辑，**重复补丁或漏补丁都会导致行为与本文不一致**；升级依赖或执行 **`idf.py update-dependencies`** 后务必 **重新检查** `bridge_eth.c` 是否仍含本仓库修改。
- 若在未验证的组合上直接套用本文结论，可能出现 **编译失败、运行仍复位 PHY、或出现新问题**，属预期风险。

4.1 什么现象

- **4G（PPP）拨号成功**，**SoftAP / Wi‑Fi 客户端**可上网，说明 modem 与 bridge 数据面基本正常。
- **PC 有线接 W5500**，本机设为 **IP `192.168.4.2`、掩码 `255.255.255.0`、网关 `192.168.4.1`**（与 IoT-Bridge 默认 ETH_LAN / SoftAP 侧常见 `192.168.4.1` 一致），但 **有线侧不通或刚通即断**。
- 串口典型顺序（节选）：
  - `bridge_modem: GOT ip event` / PPP 已连接，并打印运营商 DNS；
  - `bridge_common: [ETH_LAN     ]Name Server1: …`（把上游 DNS 下发给有线侧 DHCP/DNS）；
  - 紧接着 **`W bridge_eth: Hardware Reset Ethernet PHY`**；
  - 可选：`wifi:station: … leave` / **`bridge_wifi: STA Disconnect to the AP`**（SoftAP 下客户端被踢）；
  - 若存在 **SoftAP 与 PPP 网段冲突**，bridge 会把 SoftAP 改到 **`192.168.5.1`**，DHCP 分给手机 **`192.168.5.2`**，与 **有线仍停留在 `192.168.4.x`** 并存，属 **冲突消解后的预期分支**，不是 PC 网关填错的唯一判据。
- 其它偶发：`HUB: Root port reset failed`（USB 口/线/供电）；长时间后出现 **`pppos_input_tcpip failed with -1`**（PPP 与负载/链路稳定性相关，需在 **有线 PHY 不再被误复位** 后再观察是否仍频繁）。

4.2 如何调试 / 分析要点

1. **先确认 W5500 硬件与驱动**：参考 **条目 1、2**，能 **Link Up**、无持续 **chip ID / SPI 超时**，否则与「bridge 逻辑」无关。
2. **抓 PPP 刚连上前后 1～2 秒日志**：若 **每次** 都在 **`[ETH_LAN]Name Server1`** 后出现 **`Hardware Reset Ethernet PHY`**，高度怀疑 **软件在「仅更新 DNS」时误触发了 SPI 以太网硬件复位**。
3. **对照 `espressif/iot_bridge` 源码 `src/bridge_eth.c`**：
   - `esp_bridge_create_eth_netif(..., data_forwarding=true)` 为 **ETH_LAN** 注册  
     `esp_bridge_netif_list_add(netif, eth_netif_dhcp_status_change_cb, eth_netif_dhcp_status_change_cb)`；
   - **`dns_change_cb`** 在 `bridge_common.c` 的 `esp_bridge_update_data_forwarding_netif_dns_info()` 里以 **`NULL` 参数**调用，语义是 **「只刷新了 DNS，没有改 LAN IP」**；
   - **`dhcps_change_cb`** 在 **LAN 网段被重新分配**时带 **非空 `ip_info`** 调用；
   - **旧实现**里 `eth_netif_dhcp_status_change_cb` **无论参数是否为 NULL** 都执行 **`phy->reset_hw()`**。对 **内置 PHY** 也许历史上用于协商恢复，对 **SPI W5500** 则等价于 **整片 MAC/PHY 硬复位 → 链路必断 → PC 侧表现为「4G 转 W5500 失败」**。
4. **对照 `bridge_wifi.c` 中 `softap_netif_dhcp_status_change_cb`**：对 **`ip_info == NULL`（仅 DNS）直接 `return ESP_OK`，不复位、不 deauth**。**ETH_LAN 路径原先缺少同等保护。**

4.3 修改内容（本仓库实践）

在 **`managed_components/espressif__iot_bridge/src/bridge_eth.c`** 中调整 **`eth_netif_dhcp_status_change_cb`**（并保持与 **内置 EMAC** 行为区分）：

- **`ip_info == NULL`**：**直接 `ESP_OK`**，对应 **上游 DNS 写入 ETH_LAN**，**禁止** `reset_hw()`。
- **`CONFIG_BRIDGE_USE_SPI_ETHERNET`（W5500 等）**：**不再调用** `reset_hw()`（避免 **LAN IP 因网段冲突被重配时** 仍复位 SPI 芯片导致所有 DHCP 客户端掉线）。
- **`esp_bridge_eth_reset_phy()`**：仅用 **`#if CONFIG_BRIDGE_USE_INTERNAL_ETHERNET`** 包裹，避免 **纯 SPI 工程**出现未使用函数的告警；**内置以太网**仍可在非 NULL 分支走复位逻辑（若编译配置为 internal）。

**维护说明**：`managed_components` 在 **`idf.py update-dependencies`** 等操作后可能被 **覆盖**，若补丁丢失需 **重新合入** 或向 **esp-iot-bridge** 上游反馈该行为。

4.4 调试结果与 PC 侧建议

- **结果**：补丁后，**PPP 连上、DNS 下发到 ETH_LAN 时不再紧跟 `Hardware Reset Ethernet PHY`**，**W5500 链路可保持**，**4G → NAPT → 有线 PC** 与 **4G → Wi‑Fi** 可同时工作（具体以本机烧录验证为准）。
- **PC 静态 IP 建议**：**`192.168.4.2` / `255.255.255.0` / 网关 `192.168.4.1`**，前提是设备上 **ETH_LAN 仍为 `192.168.4.1`**；若仅使用网线、不需要热点，可在 menuconfig 中关闭 **SoftAP 数据转发**，减少与 PPP 私网的 **网段冲突与反复 deauth**（非必须）。
- **若仍见 `pppos_input_tcpip failed`**：优先检查 **USB 连接与供电**，再在 **稳定有线、无 PHY 误复位** 的前提下抓完整日志继续查 PPP/内存。

4.5 SIM 欠费 / 无流量：卡在 `Waiting for IP address` 或 `Connection lost`

- **现象**：日志停在 **`bridge_modem: Waiting for IP address`** 很久不继续；或已 **`GOT ip`** 后出现 **`esp-netif_lwip-ppp: Connection lost`**、**`Modem Disconnect from PPP Server`**。
- **原因（节选）**：上游 **`bridge_modem.c`** 里对 PPP 拿地址使用了 **`xEventGroupWaitBits(..., portMAX_DELAY)`**，在 **SIM 无数据、欠费、APN 错误** 时可能 **长时间拿不到 IP**，启动流程会一直 **堵在等 IP**；断线后 lwIP 会打 **Connection lost**，属 **数据面掉线**，与 **ETH_WAN 等其它外网** 独立。
- **本仓库对策**：在 **`main/Kconfig.projbuild` → Application modem (PPP)** 增加 **`CONFIG_BRIDGE_MODEM_WAIT_IP_TIMEOUT_MS`**（默认 **120000**），并补丁 **`managed_components/.../bridge_modem.c`**：超时则 **释放 modem netif 并返回 NULL**，**不再无限阻塞**，以太网 WAN/LAN、Wi‑Fi 等仍可继续初始化。设为 **0** 可恢复「无限等待」旧行为。
- **运行时反复掉线**：需查 **套餐/运营商/APN**；自动重拨属于 **上层策略**（本补丁只解决 **启动阶段** 死等）。

## 5.「外部网络」多选（Wi‑Fi STA / 4G / 以太网 WAN）的实际能力与限制，202604041830

5.1 背景问题

在 menuconfig **Bridge Configuration → The interface used to connect to the external network** 中，若**同时勾选**例如：

- **Use Wi‑Fi station interface to connect to the external network**（`BRIDGE_EXTERNAL_NETIF_STATION`）
- **Use 4G modem interface**（`BRIDGE_EXTERNAL_NETIF_MODEM`）
- **Use ethernet interface …**（`BRIDGE_EXTERNAL_NETIF_ETHERNET`）

是否与 **「不能同时使用」** 等价？是否存在 **WAN 负载均衡** 或 **一主多备自动切换**？

5.2 分析要点（对照 `espressif/iot_bridge` 源码与 Kconfig）

1. **Kconfig 是否禁止多选**  
   上述 STA / Modem / Ethernet 三项均为普通 **`bool`**，**没有**做成 `choice` 互斥，因此**菜单上可以全部打开**，工程**一般仍能编译**（与 SPI/SDIO 类外设不同；**SPI/SDIO** 相关宏在 `esp_bridge_config.h` 里用 **`NUM_DEFINED_MACROS > 1` 直接 `#error`**，不能多选）。

2. **「Please disable …」特指以太网角色冲突**  
   **`BRIDGE_EXTERNAL_NETIF_ETHERNET`** 的说明要求先关 **`BRIDGE_DATA_FORWARDING_NETIF_ETHERNET`**，且 Kconfig 中后者 **`depends on !BRIDGE_EXTERNAL_NETIF_ETHERNET`**。含义是：**同一块以太网在 bridge 里不能既当 WAN（外网）又当 LAN（给下游转发）**。这与「STA + 4G + 网线同时当外网」**不是同一类限制**。

3. **多选时接口会不会都被创建**  
   `bridge_common.c` 中 **`esp_bridge_create_all_netif()`** 在各宏为真时会**依次**创建 modem、STA、`ETH_WAN` 等，因此从「**是否存在多个外网 netif**」角度，**可以同时存在**。

4. **bridge 并未按「多 WAN」设计：DNS / SoftAP 侧只偏向某一个**  
   - **`bridge_wifi.c`** 里 SoftAP **`AP_START`** 时给下游推 DNS，选用 `external_netif` 的方式是**多个独立 `#if` 顺序赋值**，**后执行的覆盖前面的**。若 STA、Modem、`ETH_WAN` 全开，**最后会固定取 `WIFI_STA_DEF`**，**不代表**当前真实上网链路就是 STA，也**不会**在 PPP / 网线之间做运行时择优。  
   - **`bridge_common.c`** 中 **`dhcp_dns_before_updated_customer_cb` / `dhcp_dns_updated_customer_cb`** 使用 **`#if … #elif …`**，**编译期只保留一条分支**（例如定义了 STA 则只盯 **`WIFI_STA_DEF`**，**PPP 不在该链中**）。多外网同时启用时，**不会出现统一的「多 WAN 策略」**。

5. **无均衡、无主备**  
   **esp-iot-bridge** 不提供：多 WAN **按流量/会话分配**的负载均衡，也不提供 **主 WAN 故障自动切备 WAN** 的状态机。lwIP 侧通常也是 **单一默认路由** 语义；多网关并存时，**实际走哪条路径**易随连接顺序、事件顺序变化，**不能当作产品级多 WAN**。

5.3 结论与建议

- **结论**：多选 **≠** 官方支持的「多外网协同」；**没有**内置 WAN 均衡，**没有**内置一主多备。需要 **一条明确外网** 时，建议 **只开一个** `BRIDGE_EXTERNAL_NETIF_*`（再配 SoftAP / `ETH_LAN` 等转发口）。若要做 **冷备 / 故障切换**，需在 **`main` 或自有组件** 中实现：例如监测各接口连通性，**显式**切换默认网卡、重配 NAPT、更新下发 DNS 等。  
- **与现象日志的关系**：若见 **`Set ip info:192.168.5.1`** 与 **`IP Address:192.168.4.1`** 等交错打印，属 **网段冲突消解与多段逻辑**（参见 **条目 4**），与「多 WAN 策略」是不同维度的问题。

## 6.单 WAN 下多 LAN 并存（如 4G → Wi‑Fi + W5500 + SPI 转发），202604041945

6.1 背景问题

在 **只选一个外网**（例如仅 **`BRIDGE_EXTERNAL_NETIF_MODEM`**）的前提下，是否允许 **多个「对内转发口」同时工作**，例如：

- **SoftAP**（`BRIDGE_DATA_FORWARDING_NETIF_SOFTAP`）
- **SPI 以太网 / W5500 作 ETH_LAN**（`BRIDGE_DATA_FORWARDING_NETIF_ETHERNET`，与 **条目 4** 中 LAN 场景一致）
- **SPI 接口转发**（`BRIDGE_DATA_FORWARDING_NETIF_SPI`，IoT-Bridge 里走 **Network Adapter / `SPI_DEF`**，与 W5500 不是同一路软件栈）

6.2 分析要点（Kconfig、`esp_bridge_create_all_netif`、`esp_bridge_update_dns_info`）

1. **设计取向：单 WAN + 多 LAN 是合理组合**  
   「外网」菜单与「为其它设备提供数据转发」菜单在 Kconfig 上 **相互独立**（除 **以太网** 不能同一 MAC 既 `EXTERNAL` 又 `DATA_FORWARDING`，见 **条目 5**）。**SoftAP、USB Device Net、ETH_LAN、SPI 转发、SDIO 转发** 等数据面选项多为普通 **`bool`**，**可同时勾选**，与 **仅 Modem / 仅 STA** 等 **单 WAN** 配置兼容。

2. **创建顺序与 netif 键**  
   `bridge_common.c` 中 **`esp_bridge_create_all_netif()`** 在宏为真时依次创建：**SoftAP**（`WIFI_AP_DEF`）→ **USB 转发** → **SPI/SDIO** → **以太网（含 ETH_LAN）** → 外网侧 modem/STA/ETH_WAN 等。多 LAN 同时使能时，会存在 **多个带 DHCP 服务端或类以太网接口的 netif**，各自 **ifkey** 不同（如 **`WIFI_AP_DEF`、`ETH_LAN`、`SPI_DEF`、`USB_DEF`**），**从架构上支持并存**。

3. **DNS 同步到「所有已启用的 LAN」**  
   **`esp_bridge_update_dns_info(..., data_forwarding_netif=NULL)`** 分支里，对 **每一种** 已定义的 **`BRIDGE_DATA_FORWARDING_NETIF_*`** 分别 **`#if` 调用** `esp_bridge_update_data_forwarding_netif_dns_info`（SoftAP、SDIO、SPI、`ETH_LAN`、USB）。即 **上游 DNS 更新时，理论上会给所有已编译进来的转发口各推一份**，**不是**只服务某一个 LAN。

4. **NAPT / 转发**  
   Bridge 依赖 **`CONFIG_BRIDGE_ENABLE`** 打开的 **lwIP IP forward + IPv4 NAPT**。SoftAP / ETH / SPI 等路径在各自 **`AP_START`、拿到 IP、或接口 up** 等回调里对 **本接口地址** 调用 **`ip_napt_enable`**（或等价逻辑）。**多 LAN** 时常见形态是 **各 LAN 网段经 NAPT 共用同一条默认路由出 WAN**；具体表项规模受 **`IP_NAPT_*` 等 lwIP 配置** 与 RAM 限制，极端多连接时需调参或压测。

5. **SPI 转发 与 SDIO 不能二选一以上（编译期硬限制）**  
   `esp_bridge_config.h` 中 **`NUM_DEFINED_MACROS`** 将下列四者 **相加**：`BRIDGE_EXTERNAL_NETIF_SPI`、`BRIDGE_EXTERNAL_NETIF_SDIO`、`BRIDGE_DATA_FORWARDING_NETIF_SPI`、`BRIDGE_DATA_FORWARDING_NETIF_SDIO`；**大于 1 则 `#error`**。因此 **不能** 同时启用例如 **「SPI 外网 + SDIO 转发」** 或 **「SPI 转发 + SDIO 转发」** 等 **多种 SPI/SDIO 角色组合**；**但** **「SoftAP + ETH_LAN + 仅一种 SPI 或 SDIO 转发」** 不在此四项重复计数冲突内（SoftAP/ETH/USB 不计入该宏）。

6. **网段与冲突消解（多 LAN 时的主要坑）**  
   多个 LAN 若默认都落在 **同一私网段（如 192.168.4.0/24）**，会与 **PPP 侧网段** 或其它 LAN **冲突**。`bridge_common.c` 中 **`esp_bridge_netif_network_segment_conflict_update()`** 等逻辑会尝试 **给 SoftAP 等改段**（例如改到 **192.168.5.x**），并 **deauth、重起 DHCPS**。这是 **多 LAN + 单 WAN** 时日志里 **IP 反复变化** 的常见原因，需在 menuconfig/NVS/静态规划上 **主动错开网段**，或接受 bridge 的自动改段行为（参见 **条目 4、5**）。

7. **「SPI 接口」与 W5500 的区分**  
   - **W5500**：走 **`BRIDGE_DATA_FORWARDING_NETIF_ETHERNET`**（或 AUTO WAN/LAN），驱动为 **SPI 上的以太网 MAC**，netif 键为 **`ETH_LAN`** / **`ETH_WAN`**。  
   - **`BRIDGE_DATA_FORWARDING_NETIF_SPI`**：走 **`bridge_spi.c` + network_adapter**，面向 **另一颗 MCU 通过 SPI 挂接的从机网络适配协议**，netif 键 **`SPI_DEF`**，**不是**再给 W5500 开第二份同一类接口。二者 **可同时打开**（若硬件上既有 W5500 又有 SPI 从机网络方案），但 **引脚、SPI 主机、任务负载** 需自行避免冲突。

6.3 结论与建议

- **结论**：在 **单 WAN**（如仅 4G）下，**可以** 配置 **多 LAN 并存访问**——典型组合如本仓库 **`sdkconfig` 已存在的「Modem + SoftAP + ETH_LAN（W5500）」**；若再开 **`BRIDGE_DATA_FORWARDING_NETIF_SPI`**，在 **不满足** `NUM_DEFINED_MACROS > 1` 的前提下 **也可编译**，形成 **Wi‑Fi 客户端 + 有线 + SPI 从机侧** 多出口 LAN（是否全部在板级上实现取决于硬件）。  
- **没有**单独的「多 LAN 负载均衡」概念：下游设备各连各的接口，**共享同一条 WAN 出网**，由 **路由 + NAPT** 转发；**不是**把 WAN 带宽按 LAN 做比例分配。  
- **建议**：多 LAN 时 **规划好各接口 IP 段**，减少与 PPP 池冲突；关注 **NAPT 表与内存**；SPI/SDIO 类 **最多占上述四宏之一**；W5500 与 **SPI_DEF** 分清用途，避免误以为「两个 SPI 以太网」可随意叠加以太网转发。

## 7.单 WAN / 多 LAN 下的工作模式数量与命名（4G、W5500、Wi‑Fi STA），202604042030

7.1 前提约定

- **WAN（外网）**：同一时间 **只启用一种**，候选为  
  - ① **4G（PPP / Modem）**  
  - ② **W5500 作外网**（`BRIDGE_EXTERNAL_NETIF_ETHERNET`，netif 侧即 **`ETH_WAN`**）  
  - **③ Wi‑Fi STA 连上级路由器**（`BRIDGE_EXTERNAL_NETIF_STATION`，`WIFI_STA_DEF`）
- **LAN（对内转发）**：可同时多个，候选为  
  - **A** **Wi‑Fi SoftAP**（`BRIDGE_DATA_FORWARDING_NETIF_SOFTAP`）  
  - **B** **W5500 作内网有线**（`BRIDGE_DATA_FORWARDING_NETIF_ETHERNET`，**`ETH_LAN`**）
- 下文在 **「至少有一种 LAN」** 的网桥/路由场景下计数；若允许 **无 LAN** 的纯终端形态，见 **7.4**。

7.2 约束：同一块 W5500 不能既 WAN 又 LAN

esp-iot-bridge Kconfig：**`BRIDGE_EXTERNAL_NETIF_ETHERNET`** 与 **`BRIDGE_DATA_FORWARDING_NETIF_ETHERNET`** **互斥**（同一块以太网不能同时当外网与转发口）。因此：

- **WAN = W5500（外网）** 时，**不能再选 B（W5500 当 LAN）**；LAN 侧 **只能** 出现 **A（SoftAP）**。

Wi‑Fi 侧：**WAN = STA** 与 **LAN = SoftAP** 可同时存在（**APSTA**），与本条目不冲突。

7.3 模式数量与建议名称

**LAN 的非空子集**（A/B 两维）：**{A}、{B}、{A+B}** → **3 种**。

| WAN | 允许的 LAN 组合 | 模式数 |
|-----|-----------------|--------|
| ① 4G | {A}、{B}、{A+B} | **3** |
| ② W5500 外网 | 仅 {A}（不可含 B） | **1** |
| ③ Wi‑Fi STA | {A}、{B}、{A+B} | **3** |

**合计：3 + 1 + 3 = 7 种**（在「至少一种 LAN」前提下）。

建议 **模式名称**（文档/菜单可用）：

1. **4G → Wi‑Fi（SoftAP）**  
2. **4G → 有线（W5500 ETH_LAN）**  
3. **4G → Wi‑Fi + 有线**  
4. **Wi‑Fi STA → Wi‑Fi（SoftAP）**  
5. **Wi‑Fi STA → 有线（W5500）**  
6. **Wi‑Fi STA → Wi‑Fi + 有线**  
7. **有线 WAN（W5500）→ 仅 Wi‑Fi（SoftAP）**

7.4 其它计数方式（可选）

- 若 **不考虑「W5500 作 WAN」**（仅 **4G** 与 **Wi‑Fi STA** 两种 WAN），则每种 WAN 配 **3** 种 LAN 组合 → **2 × 3 = 6** 种。  
- 若定义 **允许「仅 WAN、无 LAN」**（不做转发），则可在每种 WAN 上各加 **1** 种「无 LAN」：在 **7.3** 的 **7** 种基础上再加 **3** 种 → **10** 种（一般 4G 网桥产品较少单独列出）。

7.5 与条目 5、6 的关系

- **条目 5**：多 WAN **同时勾选** 时 bridge **未按多 WAN 产品设计**；本条目 **刻意规定 WAN 互斥**，与产品化「一次只选一条外网」一致。  
- **条目 6**：**单 WAN + 多 LAN** 在组件层面 **支持**；本条目把 **W5500 角色互斥** 折进 **模式枚举**，避免与 Kconfig 矛盾。

## 8.ESP-IDF 5.2.6 与 W5500 通讯验证，202604051200

8.1 背景

- 工程曾以 **ESP-IDF v5.1.x** 为主调试（参见 **条目 4** 中版本表）；后续将工具链切换为 **`v5.2.6`**（以本机 `IDF_PATH` / 串口启动日志 `ESP-IDF: v5.2.6` 为准）。
- 需确认：**新版本下 W5500（SPI 以太网）是否仍能初始化并完成与对端的数据通讯**。

8.2 结论（本仓库实测）

- ESP-idf V5.2.6+esp-iot-bridge 1.0.2 配合可实现wifi转w5500，或者w5500转wifi，即便是w5500也可以获取pc共享网络的IP。

- **可以**：在 **ESP-IDF v5.2.6** 下，**W5500 可正常参与 esp_eth / esp-iot-bridge 流程**（如 **`w5500_det: W5500 detected: VERSIONR=0x04`**、**`ETH_LAN` / `ETH_WAN`** 侧 **Link 与业务通讯**），与 **5.1.x 行为一致的前提**仍是：**SPI 引脚、PHY 地址、时钟、供电与 RST** 与板级一致（参见 **条目 1～3**）。
- **换主版本后**仍需逐项核对：**menuconfig 中 `BRIDGE_ETH_SPI_*`、组件依赖、`dependencies.lock`**；若出现 **编译或 lwIP 补丁失败**，再对照 **8.3**。

8.3 esp-iot-bridge 与 lwIP 补丁（5.2.x 备忘）

- IDF **5.2.0 及以上** 自带的 **`components/lwip/lwip`** 中，`ip4_napt.c` / `lwip_napt.h` **已包含** 早期 IoT-Bridge 通过补丁加入的钩子与 **`ip_napt_table_clear` 等逻辑**，与旧版 **`.patch` 上下文不一致**，直接 **`git apply` 会失败**。
- 本仓库在 **`managed_components/espressif__iot_bridge/patches/patches.list`** 中为 **`ip4_napt.patch`**、**`lwip_napt.patch`** 增加了条件 **`idf_version < 5.2.0`**：**仅在 IDF 5.1.x 及更早** 应用；**5.2.6 上跳过**，依赖 IDF 自带实现即可。
- 若 **`managed_components` 被组件管理器覆盖**，需确认上述 **`patches.list`** 是否仍在；丢失后恢复方式：重新加入版本条件或升级 **esp-iot-bridge** 官方已适配 5.2 的版本。
