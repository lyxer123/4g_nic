（序号）1.W5500 / esp_eth 初始化失败（串口红字）的调试，202604010818

1.1 什么现象

串口出现红色错误日志，例如：

- `w5500.mac: w5500_reset: reset timeout`
- `w5500.mac: emac_w5500_init: reset w5500 failed`
- `esp_eth: esp_eth_driver_install: init mac failed`
- `esp_eth: esp_eth_start: ethernet driver handle can't be null`
- 后续周期性：`w5500_send_command: send command timeout`、`emac_w5500_receive: issue RECV command failed`、`frame read from module failed`

含义概要：通过 RST 对 W5500 做硬件复位后，芯片未在预期时间内就绪，SPI 访问无有效应答，导致 MAC 初始化失败、`eth_handle` 为空；若仍有收包路径在跑，会持续报 SPI 命令/收帧超时。属**硬件或供电/SPI 物理层**问题，不是应用层 ping 配置问题。

同一块板、布线未改，仅更换供电环境（例如家里与办公室），也可能复现或消失：两块 12V/1A 适配器经 MP1584 转 5V/3.3V 后，**纹波、瞬态负载、插头压降** 不同，会使 3.3V 在尖峰电流下短时跌落，表现为偶发 `reset timeout` / `send command timeout`。

1.2 如何调试

1. 对照 `sdkconfig`：SPI（SCK/MOSI/MISO/CS）、INT、RST 与 PCB 是否一致；RST 是否与 4G 复位等 GPIO 冲突。
2. 万用表测 W5500 附近 **3.3V** 是否稳定（是否明显低于约 3.0V 或大幅抖动）。
3. **对照实验**：只换 12V 适配器或只换供电场所，观察日志是否跟随变化。
4. 在 3.3V 靠近 ESP/W5500 处加强 **去耦（陶瓷 + 可选 47～100µF）**；缩短 12V 输入线、压紧插头。
5. 软件侧：将 `CONFIG_BRIDGE_ETH_SPI_CLOCK_MHZ` **暂时降到 10MHz 或更低**，排除布线边际下的 SPI 误码。
6. 若原理图无 MCU 控制 RST，核对是否应关闭软件 RST GPIO 或接法是否反（低有效 RSTn）。

1.3 调试结果

当前结论（分析阶段）：红色日志对应 **W5500 未正确完成硬件复位或 SPI 无应答**，优先排查 **供电瞬态/适配器差异、RST 与 SPI 接线、本地去耦与 SPI 时钟**，而非 iot_bridge 业务逻辑。板级未改仅换供电环境即可复现差异时，**电源质量**为高度可疑因素。具体哪一项起主导需按 1.2 实测后在此补充（如适配器对调、加电容、降 SPI 后的串口对比）。
