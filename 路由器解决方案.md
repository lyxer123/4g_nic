# 路由器运行时模式化改造方案

## 1. 目标

- 不依赖 `esp-iot-bridge` 的编译期“拓扑选择”来决定最终工作形态。
- 固件一次编译尽量包含所需能力（W5500、Wi-Fi STA/AP、USB Cat1/PPP 等）。
- 运行时由 Web 页面“工作模式”（存于 NVS）决定启用哪条 WAN/LAN 路径。
- 所有参数类配置（工作模式、Wi-Fi STA、W5500 WAN 等）统一持久化到 NVS。

### 1.0 产品形态：一套 `.bin`、多种电路板（BOM 有/无 W5500、有/无 4G）

你要的是 **同一份编译产物** 烧进所有 ESP32‑S3 板子，而不是「无 W5500 的板子编一版、有 4G 的再编一版」。与本节对齐的理解是：

| 层次 | 做什么 | 不做什么 |
|------|--------|----------|
| **编译期（`sdkconfig`）** | 把 **所有可能要用的链路** 都编进固件：**Wi‑Fi STA/AP**、**USB 4G（PPP）**、**单 W5500 在 AUTO 拓扑下既可能 WAN 也可能 LAN** 所需的 `esp_bridge_create_*` 与驱动。**这是在选「能力全集」，不是在 menuconfig 里提前定死「今天产品是 4G 转 Wi‑Fi 还是 STA 转 W5500」**。 | 不要为每种 BOM 做一款「只勾选一种外网、一种转发」的瘦身固件来当**唯一**发布物。 |
| **上电 / 启动** | **`system_w5500_detect_present()`**、**`system_usb_cat1_detect_present()`** 等探测：板子上 **没焊 W5500** 或 **没接 4G** 只会得到「未检测到」，**不会也不必烧不同 bin**。 | 探测结果不负责「魔法开启」未编译的协议栈；未启用对应 `CONFIG_BRIDGE_*` 的代码根本不在镜像里。 |
| **网页（连 SoftAP 进管理页）** | 用户选择 **WAN = 4G / 以太（W5500）/ Wi‑Fi STA**（Wi‑Fi 为保底能力）；后端用 **`mode_allowed`** 只展示 **当前硬件 + 当前镜像** 下合法的 `work_mode`。 | 未探测到 4G 时**不应**出现「4G 作 WAN」；未探测到 W5500 时**不应**出现「有线 WAN」或「W5500 作 LAN」。 |
| **数据面角色** | 例如：**WAN = 4G** 且板子 **有 W5500** 时，典型是 **PPP 出公网 + SoftAP 给手机/笔电 + W5500 作 ETH_LAN 给有线设备**（本仓库 profile **3**：「4G → Wi‑Fi + W5500」）。**同一颗 W5500** 在此时 **只能是 LAN**，不能同时又当 WAN。 | 「全能双 WAN 绑定线」之类超出 **单 PHY / 单 W5500** 的硬件不讨论。 |

**与 esp-iot-bridge 菜单的关系（避免误解）**

- menuconfig 里 **「连外网」「转发」** 两个菜单勾选的是 **代码是否打进镜像**，不是「出厂固定一种业务拓扑」。
- **运行时到底当 WAN 还是当 LAN**，由 **NVS `work_mode` + iot-bridge AUTO 拓扑 + Mode Manager** 决定；**无 W5500 的板子** 上只会 **跳过** `esp_bridge_create_eth_netif`，**不会因为编了 AUTO 而崩溃**。
- **「所有可能的模式都编进去」** = **对外网侧：STA + Modem +（单 W5500 用）AUTO**；**对转发侧：SoftAP 必选 + 以太侧由 AUTO 带出 ETH_LAN**（与本仓库 `sdkconfig.defaults` 一致），而不是在 Kconfig 里只勾「某一种外网接口」当作永久产品定义。

### 1.1 现状：代码、文档与网页的对照（2026-04）

**已落地（固件侧）**

- **`components/system/system_mode_manager.c`**：用 **11 条 profile** 覆盖「单 WAN × 多 LAN」组合：上行可为 **USB 4G（PPP）**、**Wi‑Fi STA**、**W5500（ETH_WAN）** 或 **无 WAN（仅热点配网）**；下行 LAN 可为 **SoftAP**、**W5500（ETH_LAN）** 或二者同时（受 **单 PHY 约束** 制约，见下）。
- **`system_mode_manager_mode_allowed()`**：在 **编译期 `CONFIG_BRIDGE_*`** 之上，再用 **`system_w5500_detect_present()`**、**`system_usb_cat1_detect_present()`** 过滤——**无 W5500 或无 USB Cat.1 时，对应 WAN/LAN 模式不会在 API 列表中出现**（与「只有 Wi‑Fi 板载必选」的产品假设一致）。
- **`system_bridge_init_netifs_from_hw()`**：在 **`esp_bridge_create_*`** 前做探测；**无 W5500 则跳过以太 netif 创建**；**`CONFIG_BRIDGE_EXTERNAL_NETIF_MODEM=y` 时 Modem netif 仍会创建**，与启动瞬间 USB 是否枚举成功解耦（与网页/模式是否允许「4G 上行」是两件事）。
- **`components/webService/src/web_service.c`**：`json_build_mode_list()` 已对前端暴露 **`mode_allowed` 后的列表**（含 `wan_type`、`lan_softap`、`lan_eth` 等字段）；另有 **`/api/...`** 可读写 `work_mode`（与 NVS `bridge_ui/work_mode` 对齐）。

**网页侧缺口（与本文目标相比）**

- **`webPage/index.html` / `app.js`** 仍以 **`working_mode`: `4g` / `router` / `ap`** 粗粒度下拉为主；保存时由 **`pick_mode_for_ui_working_mode()`** 在允许列表里**取第一个匹配项**，**无法 explicit 选择「WAN=以太 / WAN=STA / LAN 是否带有线」**。
- **目标交互**：用户先在网页选择 **WAN 类型**（Wi‑Fi / 有线 W5500 / 4G）；后端根据 **硬件探测 + `mode_allowed`** 返回 **可选的完整工作模式**（含 LAN 组合）；仅 **Wi‑Fi 必存在** 的板子不应出现依赖 W5500/4G 的选项。

**硬件与 esp-iot-bridge 的硬约束（单颗 W5500）**

- Kconfig 下 **`BRIDGE_EXTERNAL_NETIF_ETHERNET`** 与 **`BRIDGE_DATA_FORWARDING_NETIF_ETHERNET`** **互斥**：同一块 SPI 以太**不能**在编译期同时声明为「纯外网口」与「纯转发口」。
- **本仓库采用 `CONFIG_BRIDGE_NETIF_ETHERNET_AUTO_WAN_OR_LAN=y`**：由组件在运行时划分 **ETH_WAN / ETH_LAN** 角色，从而在 **单 W5500** 上仍可按模式切换「有线 WAN」或「有线 LAN」（与 **`调试.md` §7** 一致）。
- **当 WAN=W5500 时**，该 PHY 已占作外网，**不能再作为 ETH_LAN**；profile **7～9** 仅 **SoftAP** 作无线 LAN，**不允许「W5500 WAN + W5500 LAN」** 组合。

## 2. 当前痛点

- 现有实现中，部分模式可见性/可用性受 `CONFIG_BRIDGE_*` 等编译期宏影响。
- 这导致“网页可选模式”与“实际编译拓扑”耦合，无法真正运行时切换。
- `esp-iot-bridge` 原生设计包含编译期互斥项，直接依赖其拓扑开关会限制动态能力。

---

## 3. 总体设计（推荐路线）

采用“能力编译进来 + 运行时编排”的分层架构：

1. **能力层（Drivers/Caps）**  
   编译进 W5500、Wi-Fi STA/AP、USB 枚举/PPP 等基础能力。

2. **编排层（Mode Manager）**  
   新增运行时状态机，根据 NVS 中 `work_mode` 启停 netif、NAT、DHCP、DNS 和默认路由。

3. **配置层（NVS Store）**  
   将模式与参数统一收敛存储，不把 NVS 逻辑散落在各业务点。

4. **控制层（Web API）**  
   网页仅负责配置读写与“应用模式”触发，不依赖编译期拓扑判断。

---

## 4. 实施路线对比

## 路线 A（推荐，风险低）

- 保留 `esp-iot-bridge` 作为能力来源（能复用的函数继续复用）。
- 运行时模式切换由项目自有 `components/system` 编排层掌控。
- 尽量避免深改 managed component 内部逻辑。

**优点**：可控、可维护、升级冲突少。  
**缺点**：需要补一层运行时管理代码。

## 路线 B（不推荐，风险高）

- 深改 `managed_components/espressif__iot_bridge`，把编译期互斥改成可并存。
- 改动范围大，后续升级组件时冲突频繁；**Component Manager 更新可能覆盖 managed 目录**，补丁须纳入版本管理或改为自有 fork 组件。

### 路线 B 分步实施（本项目已启动）

| 阶段 | 内容 | 状态 |
|------|------|------|
| **B‑1** | 允许 **`BRIDGE_EXTERNAL_NETIF_ETHERNET`** 与 **`BRIDGE_DATA_FORWARDING_NETIF_ETHERNET`** 同时勾选（且 **`BRIDGE_NETIF_ETHERNET_AUTO_WAN_OR_LAN=n`**）；单 PHY（W5500/内建 EMAC）走与 AUTO 相同的 **共享 MAC glue**（`esp_eth_netif_glue_wrap.c`），避免双调 `esp_bridge_eth_spi_init` 炸栈。 | **已落地**：见 `managed_components/espressif__iot_bridge/LOCAL_PATCHES_README.md`（修改 `Kconfig`、`CMakeLists.txt`、`src/bridge_eth.c`）。 |
| **B‑2** | 是否允许 **AUTO 与 External+Forwarding 同时开启**（三宏叠加）及对应 `esp_bridge_create_all_netif` 语义——需逐项审 `bridge_common` / DNS / 默认路由，**当前仍保持与上游一致：AUTO 与 External+Forwarding 二选一**。 | 未做 |
| **B‑3** | 放宽 **SPI/SDIO** 多选限制、`NUM_DEFINED_MACROS` 等（若产品需要非以太扩展口）。 | 未做 |
| **B‑4** | 多 **外网 netif**（如 STA + Modem + ETH 同时存在）时 **DNS/DHCP 回调** 的优先级与可配置性。 | 未做 |

**使用 B‑1 时**：menuconfig 里二选一——要么勾选 **Ethernet AUTO WAN or LAN**，要么同时勾选 **External 以太 + Forwarding 以太**，不要三种叠加以免语义混乱。

## 6. 模式模型建议

不要只依赖硬编码编号，建议内部使用结构化模式描述：

- `wan_type`: `USB_4G | WIFI_STA | W5500`
- `lan_softap`: `true/false`
- `lan_w5500`: `true/false`
- `needs_sta_cfg`: `true/false`
- `needs_eth_wan_cfg`: `true/false`

前端仍可展示 `mode id`，但后端以结构化配置执行，便于后续扩展新模式。

### 6.1 Web：WAN 优先、再选 LAN/work mode（推荐交互）

1. **探测结果展示**（可选但建议）：页面加载时调用已有能力（或小幅扩展 API）显示 **`w5500_present`、`usb_modem_present`**，让用户理解为何某些选项灰色不可选。
2. **第一步 — 选 WAN**：`Wi‑Fi（STA 上行）` / `有线（W5500，需探测成功）` / `4G（需编译 Modem 且 USB 探测成功）`。不提供「无 WAN」作为路由主路径时，可合并进「仅 AP 配网」入口。
3. **第二步 — 选 LAN 组合**：在 **WAN 已选** 前提下，筛 profile：  
   - **仅有 SoftAP**、**仅有 ETH_LAN**（若 WAN 非 W5500 且 W5500 存在）、**SoftAP + ETH_LAN**。  
   - 若 WAN=W5500，**仅显示带 SoftAP 的 LAN**（对应 id **7～9**）。
4. **写入**：用户选定一条 profile 后，将 **`work_mode`（id）** 写入 NVS 并调用 **`system_mode_manager_apply()`**；**勿**再长期依赖「`4g`/`router`/`ap` 字符串 → 第一个匹配 profile」的模糊映射。

后端过滤规则可直接复用 **`system_mode_manager_mode_allowed()`** + 按 `wan_type` / `lan_*` 二次筛选，无需重复实现探测逻辑。

## 7. NVS 统一存储建议

## 命名空间建议

- `bridge_ui`：`work_mode`
- `wifi_cfg`：`sta_ssid`、`sta_pwd`
- `eth_wan`：`dhcp`、`ip`、`mask`、`gw`、`dns1`、`dns2`
- （可选扩展）`softap_cfg`、`lan_cfg` 等

## 原则

- 每类参数有默认值函数（缺省可恢复）。
- 每次写入必须 `commit`，并返回明确错误信息给 API。
- 配置读取失败时不崩溃，回退到安全默认策略。

---

## 8. 运行时切换流程建议

1. Web 提交模式/参数 -> 写入 NVS。  
2. 调用“应用模式”流程（即时切换）或提示重启应用。  
3. 切换时顺序建议：
   - 暂停旧 WAN/LAN 数据通路
   - 关闭旧 NAT/DHCP
   - 启动新 WAN（等待 up）
   - 启动新 LAN
   - 启用 NAT/DHCP/DNS
   - 更新状态并回报前端
4. 任一步失败 -> 回滚到上一个稳定状态并上报失败原因。

---

## 9. 风险与规避

- **路由冲突**：多 WAN/LAN 并存时默认路由混乱。  
  -> 统一 default netif 策略与切换锁。

- **NAT 时序问题**：WAN 未就绪就启 NAT 导致异常。  
  -> WAN up 事件后再启 NAT，down 事件及时清理。

- **并发冲突**：扫描、切换、拨号并发。  
  -> Mode Manager 串行队列化执行。

- **组件升级冲突**：直接深改 managed component。  
  -> 优先项目自有层封装，减少侵入。

## 12. esp-iot-bridge 编译期选项与运行时探测（技术分析）

本小节说明：为何 **menuconfig 里 “The interface used to connect to the external network” 与 “The interface used to provide network data forwarding for other devices” 不能全部关闭**；工程里如何用 **sdkconfig.defaults + 运行时探测** 达到「一块固件适配多种板载外设」；以及 **ETH_WAN 告警、默认工作模式** 的含义。

### 12.1 为什么不能把两个菜单里的选项全关掉

`esp-iot-bridge` 中 `esp_bridge_create_all_netif()` 完全由 **`#if CONFIG_BRIDGE_*`** 包起来。若在编译期把「外网接口」和「数据转发接口」相关宏全部关掉，该函数**不会创建任何 netif**（包括 SoftAP），设备将没有可用的 Wi-Fi 热点，**网页配置入口也就无从谈起**。

因此：**编译期必须在 `sdkconfig` / `sdkconfig.defaults` 中固定一套“能力全集”**（至少包含 SoftAP 转发能力，以及按需的 Modem、STA、SPI 以太网等）。这与「不必为每块板子在 menuconfig 里手选 WAN/LAN 角色」并不矛盾：角色可以交给 **运行时探测 + Mode Manager**，但 **驱动与 `esp_bridge_create_*` 是否链进固件** 仍由 Kconfig 决定。

### 12.2 本仓库的折中做法：defaults 钉死能力，运行时再建网卡

- **`sdkconfig.defaults`**：写明一套推荐能力（如 SoftAP、Wi-Fi STA、USB Modem、W5500/AUTO WAN-LAN 等），开发者日常 **以 defaults 为准**，减少在 IoT-Bridge 菜单里按板子改来改去。
- **`system_bridge_init_netifs_from_hw()`**（`components/system/system_bridge_runtime.c`）：在 **`system_hw_presence_probe_before_bridge()`** 之后调用，**按与 `esp_bridge_create_all_netif` 相同的顺序** 创建网卡，但：
  - 仅当 **`system_w5500_detect_present()`** 为真时，才调用 `esp_bridge_create_eth_netif`（含 LAN 角色；若开启 `CONFIG_BRIDGE_NETIF_ETHERNET_AUTO_WAN_OR_LAN`，还会创建 WAN 侧 eth）；
  - 若 **`CONFIG_BRIDGE_EXTERNAL_NETIF_MODEM`** 启用，**始终**调用 `esp_bridge_create_modem_netif`（与 **`system_usb_cat1_detect_present()`** 脱钩；USB Cat.1 启动枚举仅影响日志/默认模式等上层逻辑）；
  - SoftAP、STA 仍按 Kconfig 创建，保证 **至少具备热点 + 网页配置** 的能力基线（前提是 **`CONFIG_BRIDGE_DATA_FORWARDING_NETIF_SOFTAP=y`** 未被关掉）。
- **`main/app_main.c`**：使用 `system_bridge_init_netifs_from_hw()` **替代** 直接调用 `esp_bridge_create_all_netif()`。

依赖条件：

- **`CONFIG_SYSTEM_HW_DETECT_W5500=y`**：否则会跳过 W5500 相关 eth 创建（探测恒为“未检测到”）。
- **`CONFIG_SYSTEM_USB_CAT1_DETECT`**：若关闭 USB 枚举探测，则 **`system_usb_cat1_detect_present()`** 恒为假，**硬件默认工作模式**会偏向「无 USB Cat.1」分支；**不影响**已编译的 `esp_bridge_create_modem_netif` 是否调用。

### 12.3 NVS 无保存模式时：硬件默认工作模式

入口：**`system_mode_manager_apply_saved_or_hw_default()`**。

逻辑概要：

- NVS 中 **`work_mode` 存在且合法，且 `system_mode_manager_mode_allowed()` 通过** → 应用该模式；
- 否则按当前硬件与编译选项选择默认 **mode id**：
  - **Modem（编译进 PPP）+ USB Cat1 探测到 + 以太能力 + W5500 探测到** → **3**（4G + SoftAP + W5500）；
  - **仅 Modem + USB 探测到** → **1**（4G + SoftAP）；
  - **无 Modem WAN 或未探测到 USB，但有 W5500（且 eth 编译进来）** → **10**（SoftAP + W5500 LAN，无 USB WAN）；
  - **二者皆无（或不可用）** → **11**（仅 SoftAP，用于网页配网）。

新增模式定义见 **`system_mode_manager.c`**：`SYSTEM_WAN_NONE`、profile **10 / 11**。

### 12.4 日志：`ETH_WAN netif missing` 与 W5500 是否“没编译”

**不等于 W5500 没编译进工程。** 该日志通常来自 **`system_eth_uplink_debug`** 等逻辑在查找 **`ETH_WAN` 句柄**。

- 若当前构建只启用了 **数据转发以太网（如 `ETH_LAN`）**，而没有 **`CONFIG_BRIDGE_EXTERNAL_NETIF_ETHERNET`** 或 **`CONFIG_BRIDGE_NETIF_ETHERNET_AUTO_WAN_OR_LAN`** 带来的 WAN 侧 netif，则 **`ETH_WAN` 不存在**，就会打印此类告警。
- **W5500 仍可已作为 `ETH_LAN`（或 AUTO 拓扑中的 LAN/WAN 之一）存在**；需结合 `sdkconfig` 里 IoT-Bridge 以太网相关项一起理解，而不是仅凭一条日志判断驱动未包含。

### 12.5 Wi-Fi STA、DHCP 与 `esp_netif_free_rx_buffer` 崩溃（背景）

早期问题现象：**STA 关联后出现 `dhcp client start failed`，随后在 `esp_netif_free_rx_buffer` 发生 `LoadProhibited`（如对 NULL 偏移 `0x34`）**。根因方向包括：

- **`wlanif_input` 异常路径**在 **`netif->state` 为空** 时仍调用 `esp_netif_free_rx_buffer`；
- **STA 网卡生命周期与 IoT-Bridge / Web 侧创建顺序不一致**（例如在编译期未启用 `BRIDGE_EXTERNAL_NETIF_STATION` 时较晚 `create_default_wifi_sta`）、或 **清除 STA 配置时破坏网卡状态** 等。

已通过：**在 defaults 中启用 `CONFIG_BRIDGE_EXTERNAL_NETIF_STATION`**、规范 **`uri_wifi_clear`** 行为、以及 **LWIP DHCP 选项与 STA 认证阈值** 等修复路径缓解；详细排障应以 **当前分支代码 + 完整串口 log** 为准。

### 12.6 使用注意（避免误配）

- 若在 **menuconfig 中关闭** **`BRIDGE_DATA_FORWARDING_NETIF_SOFTAP`**，则编译期无 SoftAP，**无法满足“至少热点配网页”** 的产品需求。
- 工作副本 **`sdkconfig`** 若与 **`sdkconfig.defaults` 长期不一致，建议 **`idf.py fullclean` 后重配**，避免“文档写了探测逻辑，实际镜像仍是旧 Kconfig”的偏差。

### 12.7 澄清：仅勾选「Wi‑Fi AP + Wi‑Fi STA」、不选 ETH/4G，能否靠探测 + Web 模式使用 W5500/4G？

**不能**若指望在 IoT-Bridge 两个菜单里**只打开** SoftAP（数据转发）与 **Wi‑Fi Station（外网）**，同时把 **以太网、USB Modem/4G** 相关 **`CONFIG_BRIDGE_*` 全部关掉**，再依靠 **W5500 / USB 硬件探测** 与 **网页工作模式** 去“启用” 4G 或 W5500：

| 层面 | 作用 |
|------|------|
| **编译期（Kconfig）** | 决定 `esp_bridge_create_modem_netif`、`esp_bridge_create_eth_netif` 及配套驱动是否**链进固件**。关掉后，对应符号在链接产物中**不存在**，不是“默认关闭、运行时再打开”的开关。 |
| **运行期探测** | 只能在这些**已编译进来的能力**上，决定**是否调用** `create_*`（例如无 W5500 则不建 `ETH_*`），**不能**让从未编译的 PPP/以太路径凭空出现。 |
| **Web 选模式** | 在已存在的 netif 与已编译策略上编排（拨号、DHCP、STA、默认路由等）。若固件未编 modem/eth，**只能禁用或提示不支持**，无法真正走 4G/W5500 网桥。 |

**推荐分工（与本仓库路线一致）**：

1. **sdkconfig / `sdkconfig.defaults`**：为产品线保留**能力全集**（SoftAP、STA、按需的 **`BRIDGE_EXTERNAL_NETIF_MODEM`**、W5500/以太对应的 **`BRIDGE_DATA_FORWARDING_NETIF_ETHERNET` / `BRIDGE_NETIF_ETHERNET_AUTO_WAN_OR_LAN`** 等），保证 `esp_bridge_create_*` 与驱动在镜像内可用。  
2. **启动时**：`system_hw_presence_probe_before_bridge()` + **`system_bridge_init_netifs_from_hw()`** —— **无硬件则不创建**对应 eth/modem netif，减少空挂接口与无效初始化。  
3. **`system_mode_manager`** + **Web**：在用户可选的模式集合内，结合 **`mode_allowed()`**（硬件未见到则相关 WAN 模式不可用）应用配置。

**一句话**：**“有没有 4G/W5500 网桥代码”靠编译；“建不建、用不用这条链路”靠探测与 Mode Manager。** 二者不可互换。

## 13. 结论

建议采用**路线 A（运行时编排层）**：  
在不深度侵入 `esp-iot-bridge` managed component 的前提下，实现“编译能力全集 + Web 运行时模式选择 + NVS 参数持久化”的目标，长期维护成本最低、可扩展性最好。  
**补充**：通过 **`system_bridge_init_netifs_from_hw` + `system_mode_manager_apply_saved_or_hw_default`**，在保持 IoT-Bridge **编译期能力集** 的前提下，对外设做 **运行时探测与默认模式落盘**，与上述结论一致。  

面向 **「WAN=Wi‑Fi / W5500 / 4G 三选一、LAN=SoftAP ± W5500」** 的产品形态：**menuconfig 勾选策略见 §15**；**网页侧由粗粒度 `working_mode` 升级为 WAN 优先、再选 LAN 的交互见 §6.1**，并与既有 **`json_build_mode_list` + `mode_allowed`** 对齐。

### 14.2 拓扑与已通过项（4G + Wi‑Fi + W5500）

在 **USB Cat.1 经 PPP 作外网**、**SoftAP 提供 Wi‑Fi 接入**、**W5500 作数据转发以太（典型为 ETH_LAN）** 的目标拓扑下，与本仓库 `sdkconfig.defaults` 能力集及板级接线一致时，实测确认：

- **4G（PPP）**：可完成拨号并出现 **`GOT ip event`** 等连接日志，上游 DNS 可被 bridge 同步到转发侧网卡。
- **Wi‑Fi（SoftAP）**：热点与 **NAPT** 可用，无线终端经 SoftAP 访问外网路径可通；若 **SoftAP 网段与 PPP 私网冲突**，bridge 将 SoftAP 迁至 **`192.168.5.1/24`** 等属**预期**行为（详见 **`调试.md` §4**）。
- **W5500**：在 **SPI 引脚、时钟、供电与 RST** 正常前提下，可出现 **`w5500_det: W5500 detected: VERSIONR=0x04`**，**`ETH_LAN`**（及按 Kconfig 存在的 **`ETH_WAN`**）**Link 与业务通讯**可参与 esp_eth / IoT-Bridge 流程；同一版本栈下 **「Wi‑Fi ↔ W5500」单转**亦已验证可通（**`调试.md` §8**）。
- **组合结论**：**ESP-IDF v5.2.6 + esp-iot-bridge 1.0.2** 下，**4G 转 Wi‑Fi 与 W5500 有线 LAN 同开**的数据面在本仓库调试中**测试通过**。历史上 **PPP 刚连上后误触发 W5500 PHY 硬件复位** 等问题属于 **`bridge_eth.c` 与上游时序**范畴，**`调试.md` §4** 记有分析与改法；修复后不影响本附录对「该版本栈可工作」的定性记录。

### 14.3 复现与文档分工

- 编译期须在 **`sdkconfig` / `sdkconfig.defaults`** 中固定 **Modem（`CONFIG_BRIDGE_EXTERNAL_NETIF_MODEM`）、SoftAP 转发、W5500/以太相关 `CONFIG_BRIDGE_*`** 等能力，使对应 **`esp_bridge_create_*`** 链进固件；**`CONFIG_SYSTEM_HW_DETECT_W5500`** 等与 W5500 探测相关的项需与板子一致。
- **`ETH_WAN netif missing`** 类日志多与**当前未创建 WAN 侧以太 netif** 的配置有关，**不等于 W5500 未编译**；说明见本文 **§12.4**。
- 启用 **`CONFIG_BRIDGE_EXTERNAL_NETIF_MODEM`** 时，**modem（PPP）网卡**会在 **`system_bridge_init_netifs_from_hw()`** 中创建，**与启动阶段 USB Cat.1 枚举是否报 `num_devices=0` 无绑定关系**（见 **`components/system/system_bridge_runtime.c`**）。
- 深度排障、串口片段与补丁上下文仍以 **`调试.md`** 为主；本附录仅作 **`路由器解决方案.md`** 侧的**版本对齐 + 通过结论**索引。

## 15. esp-iot-bridge 两个菜单如何勾选（支持「运行时选 WAN」的路由器）

设计原则：**menuconfig 勾选的是「固件里有没有这类驱动/创建函数」**；**网页选的是「当前用哪条 WAN、哪几个 LAN」**。没有_compile 进的接口，运行时再勾选也无效（见 **§12.7**）。

### 15.1 菜单 A：The interface used to connect to the external network

**若采用 §1.0「单一通用 `.bin`」**：下列 **能力向** 选项应对 **所有** BOM 组合 **统一勾选**（无 4G / 无 W5500 的板子靠 **探测 + 不创建 netif + `mode_allowed`** 收口，而不是少编一种能力）。

| 选项 | 单一通用固件（推荐，§1.0） | 仅做第二条「极小镜像」产品线时可否关掉 | 说明 |
|------|---------------------------|----------------------------------------|------|
| **Wi‑Fi Station** | **✅ 必选** | — | STA 作上行 WAN；与芯片内建 Wi‑Fi 一致。 |
| **4G modem** | **✅ 勾选** | 可关（少 PPP/modem 代码体积） | 勾选才有 **`esp_bridge_create_modem_netif`**。**无 4G 模块** 时 USB 探测为假 → 网页不出现 4G WAN；镜像略大是可接受的代价。 |
| **Ethernet（固定 External）** | **❌ 不选**（单 W5500 + 要 WAN/LAN 运行时切换时） | 仅当你做「W5500 永不当 LAN」的专用固件时可考虑 | 与 **Forwarding 以太** 互斥；与 **AUTO** 二选一风格产品。 |
| **Ethernet AUTO WAN or LAN** | **✅ 必选** | — | **单 W5500** 在「4G（或 STA）作 WAN 时 W5500 当 LAN」「W5500 单独作 WAN」等之间切换，靠 AUTO；**无 W5500 板子** 上探测失败则 **跳过 eth 创建**。 |

**归纳**

- **一套 bin 覆盖 §1.0**：**Station ✅ + Modem ✅ + AUTO WAN/LAN ✅**；**不要**只勾选「某一种外网」当全系列产品编译策略。  
- **无 W5500 / 无 4G** 时：**不创建对应 netif** 或 **有 netif 但模式不展示**，由 **`system_bridge_init_netifs_from_hw`** 与 **`mode_allowed`** 处理。

### 15.2 菜单 B：The interface used to provide network data forwarding for other devices

| 选项 | 单一通用固件（§1.0） | 说明 |
|------|---------------------|------|
| **Wi‑Fi SoftAP** | **✅ 必须勾选** | 进网页配网、无线 LAN；关闭则无热点（见 **§12.1**）。 |
| **Ethernet（转发）** | **❌ 不单独勾选** | 与 **§1.0 + AUTO** 策略一致：由 **`BRIDGE_NETIF_ETHERNET_AUTO_WAN_OR_LAN`** 在 **`system_bridge_init_netifs_from_hw`** 中按需创建 **ETH_LAN**（有 W5500 时）。若 **不用 AUTO**、且 W5500 **永只做 LAN**，才可改为单独 **Forwarding Ethernet**（并避免与固定 External 以太互斥）。 |
| **USB / SPI / SDIO 等转发** | **默认不勾** | 路由器主线不必为「换 WAN」全开；避免与 **§12.7** 互斥组合。 |

### 15.3 「如何有效勾选」一句话

- **§1.0 单一 `.bin`**：**菜单 A：`Station` + `Modem` + `AUTO WAN/LAN` 全开**；**菜单 B：`SoftAP` 全开**；**Forwarding 以太不单独勾**，交给 AUTO。无 W5500/无 4G 的板子：**探测 + 跳过创建 + 网页不展示** 对应模式。  
- **另一条「极简 sku」**（刻意不做 4G、不做有线）：才可关掉 **Modem / AUTO**，单独发布更小镜像；**与「全 BOM 一套 bin」是不同产品策略**，不要与 §1.0 混为一谈。
