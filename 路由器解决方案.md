# 路由器运行时模式化改造方案

## 1. 目标

- 不依赖 `esp-iot-bridge` 的编译期“拓扑选择”来决定最终工作形态。
- 固件一次编译尽量包含所需能力（W5500、Wi-Fi STA/AP、USB Cat1/PPP 等）。
- 运行时由 Web 页面“工作模式”（存于 NVS）决定启用哪条 WAN/LAN 路径。
- 所有参数类配置（工作模式、Wi-Fi STA、W5500 WAN 等）统一持久化到 NVS。

---

## 2. 当前痛点

- 现有实现中，部分模式可见性/可用性受 `CONFIG_BRIDGE_*` 等编译期宏影响。
- 这导致“网页可选模式”与“实际编译拓扑”耦合，无法真正运行时切换。
- `esp-iot-bridge` 原生设计包含编译期互斥项，直接依赖其拓扑开关会限制动态能力。

---

## 3. 总体设计（推荐路线）

采用“能力编译进来 + 运行时编排”的分层架构：

1. **能力层（Drivers/Caps）**  
   编译进 W5500、Wi-Fi STA/AP、USB 枚举/PPP 等基础能力。

2. **编排层（Mode Manager）**  
   新增运行时状态机，根据 NVS 中 `work_mode` 启停 netif、NAT、DHCP、DNS 和默认路由。

3. **配置层（NVS Store）**  
   将模式与参数统一收敛存储，不把 NVS 逻辑散落在各业务点。

4. **控制层（Web API）**  
   网页仅负责配置读写与“应用模式”触发，不依赖编译期拓扑判断。

---

## 4. 实施路线对比

## 路线 A（推荐，风险低）

- 保留 `esp-iot-bridge` 作为能力来源（能复用的函数继续复用）。
- 运行时模式切换由项目自有 `components/system` 编排层掌控。
- 尽量避免深改 managed component 内部逻辑。

**优点**：可控、可维护、升级冲突少。  
**缺点**：需要补一层运行时管理代码。

## 路线 B（不推荐，风险高）

- 深改 `managed_components/espressif__iot_bridge`，把编译期互斥改成可并存。
- 改动范围大，后续升级组件时冲突频繁。

---

## 5. 建议改动文件清单

## 5.1 新增（核心）

- `components/system/system_mode_manager.c/.h`  
  统一模式状态机（读取 mode，切换 WAN/LAN，错误回滚）。

- `components/system/system_mode_apply.c/.h`  
  各模式具体应用逻辑（启停 Wi-Fi STA/AP、ETH、PPP、NAT、DHCP）。

- `components/system/system_netif_registry.c/.h`  
  统一 netif 获取/创建/销毁，避免重复创建和句柄失效。

- `components/system/system_napt_ctrl.c/.h`  
  NAT/NAPT 启停、时序、容错封装。

- （建议）`components/system/system_config_store.c/.h`  
  统一封装 NVS 命名空间、键、读写与默认值策略。

## 5.2 修改（已有）

- `main` 启动入口（如 `app_main.c`）  
  调整启动顺序：硬件探测 -> 读取 NVS -> 应用模式 -> 启动 Web。

- `components/webService/src/web_service.c`  
  扩展/统一 API：`/api/mode`、`/api/wifi`、`/api/eth_wan`、（可选）`/api/mode/apply`。

- `webPage/index.html`、`webPage/js/app.js`  
  由后端返回模式列表；不同模式下显示对应参数卡片（STA/W5500 WAN）。

- `sdkconfig.defaults`  
  改为“能力全开”导向（在硬件资源可承受范围内）。

---

## 6. 模式模型建议

不要只依赖硬编码编号，建议内部使用结构化模式描述：

- `wan_type`: `USB_4G | WIFI_STA | W5500`
- `lan_softap`: `true/false`
- `lan_w5500`: `true/false`
- `needs_sta_cfg`: `true/false`
- `needs_eth_wan_cfg`: `true/false`

前端仍可展示 `mode id`，但后端以结构化配置执行，便于后续扩展新模式。

---

## 7. NVS 统一存储建议

## 命名空间建议

- `bridge_ui`：`work_mode`
- `wifi_cfg`：`sta_ssid`、`sta_pwd`
- `eth_wan`：`dhcp`、`ip`、`mask`、`gw`、`dns1`、`dns2`
- （可选扩展）`softap_cfg`、`lan_cfg` 等

## 原则

- 每类参数有默认值函数（缺省可恢复）。
- 每次写入必须 `commit`，并返回明确错误信息给 API。
- 配置读取失败时不崩溃，回退到安全默认策略。

---

## 8. 运行时切换流程建议

1. Web 提交模式/参数 -> 写入 NVS。  
2. 调用“应用模式”流程（即时切换）或提示重启应用。  
3. 切换时顺序建议：
   - 暂停旧 WAN/LAN 数据通路
   - 关闭旧 NAT/DHCP
   - 启动新 WAN（等待 up）
   - 启动新 LAN
   - 启用 NAT/DHCP/DNS
   - 更新状态并回报前端
4. 任一步失败 -> 回滚到上一个稳定状态并上报失败原因。

---

## 9. 风险与规避

- **路由冲突**：多 WAN/LAN 并存时默认路由混乱。  
  -> 统一 default netif 策略与切换锁。

- **NAT 时序问题**：WAN 未就绪就启 NAT 导致异常。  
  -> WAN up 事件后再启 NAT，down 事件及时清理。

- **并发冲突**：扫描、切换、拨号并发。  
  -> Mode Manager 串行队列化执行。

- **组件升级冲突**：直接深改 managed component。  
  -> 优先项目自有层封装，减少侵入。

---

## 10. 改动量评估

- **中到大改动**（核心在运行时编排层）。
- 预计：
  - 新增代码约 `800~1500` 行
  - 修改既有代码约 `300~700` 行

---

## 11. 分阶段落地计划

## Phase 1（先可用）

- NVS 统一化（mode/STA/eth_wan）
- Web API 与前端联动完善
- 启动时按保存模式应用（允许“重启后生效”）

## Phase 2（体验增强）

- 支持无重启模式切换
- 状态回报与失败回滚

## Phase 3（稳定性）

- 异常恢复（拔线/掉电/模块消失）
- 压测与边界验证

---

## 12. 结论

建议采用**路线 A（运行时编排层）**：  
在不深度侵入 `esp-iot-bridge` managed component 的前提下，实现“编译能力全集 + Web 运行时模式选择 + NVS 参数持久化”的目标，长期维护成本最低、可扩展性最好。
