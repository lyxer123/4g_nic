# 路由器运行时模式化改造方案

## 1. 目标

- 不依赖 `esp-iot-bridge` 的编译期“拓扑选择”来决定最终工作形态。
- 固件一次编译尽量包含所需能力（W5500、Wi-Fi STA/AP、USB Cat1/PPP 等）。
- 运行时由 Web 页面“工作模式”（存于 NVS）决定启用哪条 WAN/LAN 路径。
- 所有参数类配置（工作模式、Wi-Fi STA、W5500 WAN 等）统一持久化到 NVS。

---

## 2. 当前痛点

- 现有实现中，部分模式可见性/可用性受 `CONFIG_BRIDGE_*` 等编译期宏影响。
- 这导致“网页可选模式”与“实际编译拓扑”耦合，无法真正运行时切换。
- `esp-iot-bridge` 原生设计包含编译期互斥项，直接依赖其拓扑开关会限制动态能力。

---

## 3. 总体设计（推荐路线）

采用“能力编译进来 + 运行时编排”的分层架构：

1. **能力层（Drivers/Caps）**  
   编译进 W5500、Wi-Fi STA/AP、USB 枚举/PPP 等基础能力。

2. **编排层（Mode Manager）**  
   新增运行时状态机，根据 NVS 中 `work_mode` 启停 netif、NAT、DHCP、DNS 和默认路由。

3. **配置层（NVS Store）**  
   将模式与参数统一收敛存储，不把 NVS 逻辑散落在各业务点。

4. **控制层（Web API）**  
   网页仅负责配置读写与“应用模式”触发，不依赖编译期拓扑判断。

---

## 4. 实施路线对比

## 路线 A（推荐，风险低）

- 保留 `esp-iot-bridge` 作为能力来源（能复用的函数继续复用）。
- 运行时模式切换由项目自有 `components/system` 编排层掌控。
- 尽量避免深改 managed component 内部逻辑。

**优点**：可控、可维护、升级冲突少。  
**缺点**：需要补一层运行时管理代码。

## 路线 B（不推荐，风险高）

- 深改 `managed_components/espressif__iot_bridge`，把编译期互斥改成可并存。
- 改动范围大，后续升级组件时冲突频繁。

---

## 5. 建议改动文件清单

## 5.1 新增（核心）

- `components/system/system_mode_manager.c/.h`  
  统一模式状态机（读取 mode，切换 WAN/LAN，错误回滚）。

- `components/system/system_mode_apply.c/.h`  
  各模式具体应用逻辑（启停 Wi-Fi STA/AP、ETH、PPP、NAT、DHCP）。

- `components/system/system_netif_registry.c/.h`  
  统一 netif 获取/创建/销毁，避免重复创建和句柄失效。

- `components/system/system_napt_ctrl.c/.h`  
  NAT/NAPT 启停、时序、容错封装。

- （建议）`components/system/system_config_store.c/.h`  
  统一封装 NVS 命名空间、键、读写与默认值策略。

## 5.2 修改（已有）

- `main` 启动入口（如 `app_main.c`）  
  调整启动顺序：硬件探测 -> 读取 NVS -> 应用模式 -> 启动 Web。

- `components/webService/src/web_service.c`  
  扩展/统一 API：`/api/mode`、`/api/wifi`、`/api/eth_wan`、（可选）`/api/mode/apply`。

- `webPage/index.html`、`webPage/js/app.js`  
  由后端返回模式列表；不同模式下显示对应参数卡片（STA/W5500 WAN）。

- `sdkconfig.defaults`  
  改为“能力全开”导向（在硬件资源可承受范围内）。

---

## 6. 模式模型建议

不要只依赖硬编码编号，建议内部使用结构化模式描述：

- `wan_type`: `USB_4G | WIFI_STA | W5500`
- `lan_softap`: `true/false`
- `lan_w5500`: `true/false`
- `needs_sta_cfg`: `true/false`
- `needs_eth_wan_cfg`: `true/false`

前端仍可展示 `mode id`，但后端以结构化配置执行，便于后续扩展新模式。

---

## 7. NVS 统一存储建议

## 命名空间建议

- `bridge_ui`：`work_mode`
- `wifi_cfg`：`sta_ssid`、`sta_pwd`
- `eth_wan`：`dhcp`、`ip`、`mask`、`gw`、`dns1`、`dns2`
- （可选扩展）`softap_cfg`、`lan_cfg` 等

## 原则

- 每类参数有默认值函数（缺省可恢复）。
- 每次写入必须 `commit`，并返回明确错误信息给 API。
- 配置读取失败时不崩溃，回退到安全默认策略。

---

## 8. 运行时切换流程建议

1. Web 提交模式/参数 -> 写入 NVS。  
2. 调用“应用模式”流程（即时切换）或提示重启应用。  
3. 切换时顺序建议：
   - 暂停旧 WAN/LAN 数据通路
   - 关闭旧 NAT/DHCP
   - 启动新 WAN（等待 up）
   - 启动新 LAN
   - 启用 NAT/DHCP/DNS
   - 更新状态并回报前端
4. 任一步失败 -> 回滚到上一个稳定状态并上报失败原因。

---

## 9. 风险与规避

- **路由冲突**：多 WAN/LAN 并存时默认路由混乱。  
  -> 统一 default netif 策略与切换锁。

- **NAT 时序问题**：WAN 未就绪就启 NAT 导致异常。  
  -> WAN up 事件后再启 NAT，down 事件及时清理。

- **并发冲突**：扫描、切换、拨号并发。  
  -> Mode Manager 串行队列化执行。

- **组件升级冲突**：直接深改 managed component。  
  -> 优先项目自有层封装，减少侵入。

---

## 10. 改动量评估

- **中到大改动**（核心在运行时编排层）。
- 预计：
  - 新增代码约 `800~1500` 行
  - 修改既有代码约 `300~700` 行

## 11. 分阶段落地计划

## Phase 1（先可用）

- NVS 统一化（mode/STA/eth_wan）
- Web API 与前端联动完善
- 启动时按保存模式应用（允许“重启后生效”）

## Phase 2（体验增强）

- 支持无重启模式切换
- 状态回报与失败回滚

## Phase 3（稳定性）

- 异常恢复（拔线/掉电/模块消失）
- 压测与边界验证

## 12. esp-iot-bridge 编译期选项与运行时探测（技术分析）

本小节说明：为何 **menuconfig 里 “The interface used to connect to the external network” 与 “The interface used to provide network data forwarding for other devices” 不能全部关闭**；工程里如何用 **sdkconfig.defaults + 运行时探测** 达到「一块固件适配多种板载外设」；以及 **ETH_WAN 告警、默认工作模式** 的含义。

### 12.1 为什么不能把两个菜单里的选项全关掉

`esp-iot-bridge` 中 `esp_bridge_create_all_netif()` 完全由 **`#if CONFIG_BRIDGE_*`** 包起来。若在编译期把「外网接口」和「数据转发接口」相关宏全部关掉，该函数**不会创建任何 netif**（包括 SoftAP），设备将没有可用的 Wi-Fi 热点，**网页配置入口也就无从谈起**。

因此：**编译期必须在 `sdkconfig` / `sdkconfig.defaults` 中固定一套“能力全集”**（至少包含 SoftAP 转发能力，以及按需的 Modem、STA、SPI 以太网等）。这与「不必为每块板子在 menuconfig 里手选 WAN/LAN 角色」并不矛盾：角色可以交给 **运行时探测 + Mode Manager**，但 **驱动与 `esp_bridge_create_*` 是否链进固件** 仍由 Kconfig 决定。

### 12.2 本仓库的折中做法：defaults 钉死能力，运行时再建网卡

- **`sdkconfig.defaults`**：写明一套推荐能力（如 SoftAP、Wi-Fi STA、USB Modem、W5500/AUTO WAN-LAN 等），开发者日常 **以 defaults 为准**，减少在 IoT-Bridge 菜单里按板子改来改去。
- **`system_bridge_init_netifs_from_hw()`**（`components/system/system_bridge_runtime.c`）：在 **`system_hw_presence_probe_before_bridge()`** 之后调用，**按与 `esp_bridge_create_all_netif` 相同的顺序** 创建网卡，但：
  - 仅当 **`system_w5500_detect_present()`** 为真时，才调用 `esp_bridge_create_eth_netif`（含 LAN 角色；若开启 `CONFIG_BRIDGE_NETIF_ETHERNET_AUTO_WAN_OR_LAN`，还会创建 WAN 侧 eth）；
  - 仅当 **`system_usb_cat1_detect_present()`** 为真时，才调用 `esp_bridge_create_modem_netif`；
  - SoftAP、STA 仍按 Kconfig 创建，保证 **至少具备热点 + 网页配置** 的能力基线（前提是 **`CONFIG_BRIDGE_DATA_FORWARDING_NETIF_SOFTAP=y`** 未被关掉）。
- **`main/app_main.c`**：使用 `system_bridge_init_netifs_from_hw()` **替代** 直接调用 `esp_bridge_create_all_netif()`。

依赖条件：

- **`CONFIG_SYSTEM_HW_DETECT_W5500=y`**：否则会跳过 W5500 相关 eth 创建（探测恒为“未检测到”）。
- **`CONFIG_SYSTEM_USB_CAT1_DETECT`**：若关闭 USB 枚举探测，则 Modem 侧会被视为不存在，默认工作模式会偏向「仅 SoftAP」或「仅 W5500」分支。

### 12.3 NVS 无保存模式时：硬件默认工作模式

入口：**`system_mode_manager_apply_saved_or_hw_default()`**。

逻辑概要：

- NVS 中 **`work_mode` 存在且合法，且 `system_mode_manager_mode_allowed()` 通过** → 应用该模式；
- 否则按当前硬件与编译选项选择默认 **mode id**：
  - **Modem（编译进 PPP）+ USB Cat1 探测到 + 以太能力 + W5500 探测到** → **3**（4G + SoftAP + W5500）；
  - **仅 Modem + USB 探测到** → **1**（4G + SoftAP）；
  - **无 Modem WAN 或未探测到 USB，但有 W5500（且 eth 编译进来）** → **10**（SoftAP + W5500 LAN，无 USB WAN）；
  - **二者皆无（或不可用）** → **11**（仅 SoftAP，用于网页配网）。

新增模式定义见 **`system_mode_manager.c`**：`SYSTEM_WAN_NONE`、profile **10 / 11**。

### 12.4 日志：`ETH_WAN netif missing` 与 W5500 是否“没编译”

**不等于 W5500 没编译进工程。** 该日志通常来自 **`system_eth_uplink_debug`** 等逻辑在查找 **`ETH_WAN` 句柄**。

- 若当前构建只启用了 **数据转发以太网（如 `ETH_LAN`）**，而没有 **`CONFIG_BRIDGE_EXTERNAL_NETIF_ETHERNET`** 或 **`CONFIG_BRIDGE_NETIF_ETHERNET_AUTO_WAN_OR_LAN`** 带来的 WAN 侧 netif，则 **`ETH_WAN` 不存在**，就会打印此类告警。
- **W5500 仍可已作为 `ETH_LAN`（或 AUTO 拓扑中的 LAN/WAN 之一）存在**；需结合 `sdkconfig` 里 IoT-Bridge 以太网相关项一起理解，而不是仅凭一条日志判断驱动未包含。

### 12.5 Wi-Fi STA、DHCP 与 `esp_netif_free_rx_buffer` 崩溃（背景）

早期问题现象：**STA 关联后出现 `dhcp client start failed`，随后在 `esp_netif_free_rx_buffer` 发生 `LoadProhibited`（如对 NULL 偏移 `0x34`）**。根因方向包括：

- **`wlanif_input` 异常路径**在 **`netif->state` 为空** 时仍调用 `esp_netif_free_rx_buffer`；
- **STA 网卡生命周期与 IoT-Bridge / Web 侧创建顺序不一致**（例如在编译期未启用 `BRIDGE_EXTERNAL_NETIF_STATION` 时较晚 `create_default_wifi_sta`）、或 **清除 STA 配置时破坏网卡状态** 等。

已通过：**在 defaults 中启用 `CONFIG_BRIDGE_EXTERNAL_NETIF_STATION`**、规范 **`uri_wifi_clear`** 行为、以及 **LWIP DHCP 选项与 STA 认证阈值** 等修复路径缓解；详细排障应以 **当前分支代码 + 完整串口 log** 为准。

### 12.6 使用注意（避免误配）

- 若在 **menuconfig 中关闭** **`BRIDGE_DATA_FORWARDING_NETIF_SOFTAP`**，则编译期无 SoftAP，**无法满足“至少热点配网页”** 的产品需求。
- 工作副本 **`sdkconfig`** 若与 **`sdkconfig.defaults` 长期不一致，建议 **`idf.py fullclean` 后重配**，避免“文档写了探测逻辑，实际镜像仍是旧 Kconfig”的偏差。

### 12.7 澄清：仅勾选「Wi‑Fi AP + Wi‑Fi STA」、不选 ETH/4G，能否靠探测 + Web 模式使用 W5500/4G？

**不能**若指望在 IoT-Bridge 两个菜单里**只打开** SoftAP（数据转发）与 **Wi‑Fi Station（外网）**，同时把 **以太网、USB Modem/4G** 相关 **`CONFIG_BRIDGE_*` 全部关掉**，再依靠 **W5500 / USB 硬件探测** 与 **网页工作模式** 去“启用” 4G 或 W5500：

| 层面 | 作用 |
|------|------|
| **编译期（Kconfig）** | 决定 `esp_bridge_create_modem_netif`、`esp_bridge_create_eth_netif` 及配套驱动是否**链进固件**。关掉后，对应符号在链接产物中**不存在**，不是“默认关闭、运行时再打开”的开关。 |
| **运行期探测** | 只能在这些**已编译进来的能力**上，决定**是否调用** `create_*`（例如无 W5500 则不建 `ETH_*`），**不能**让从未编译的 PPP/以太路径凭空出现。 |
| **Web 选模式** | 在已存在的 netif 与已编译策略上编排（拨号、DHCP、STA、默认路由等）。若固件未编 modem/eth，**只能禁用或提示不支持**，无法真正走 4G/W5500 网桥。 |

**推荐分工（与本仓库路线一致）**：

1. **sdkconfig / `sdkconfig.defaults`**：为产品线保留**能力全集**（SoftAP、STA、按需的 **`BRIDGE_EXTERNAL_NETIF_MODEM`**、W5500/以太对应的 **`BRIDGE_DATA_FORWARDING_NETIF_ETHERNET` / `BRIDGE_NETIF_ETHERNET_AUTO_WAN_OR_LAN`** 等），保证 `esp_bridge_create_*` 与驱动在镜像内可用。  
2. **启动时**：`system_hw_presence_probe_before_bridge()` + **`system_bridge_init_netifs_from_hw()`** —— **无硬件则不创建**对应 eth/modem netif，减少空挂接口与无效初始化。  
3. **`system_mode_manager`** + **Web**：在用户可选的模式集合内，结合 **`mode_allowed()`**（硬件未见到则相关 WAN 模式不可用）应用配置。

**一句话**：**“有没有 4G/W5500 网桥代码”靠编译；“建不建、用不用这条链路”靠探测与 Mode Manager。** 二者不可互换。

---

## 13. 结论

建议采用**路线 A（运行时编排层）**：  
在不深度侵入 `esp-iot-bridge` managed component 的前提下，实现“编译能力全集 + Web 运行时模式选择 + NVS 参数持久化”的目标，长期维护成本最低、可扩展性最好。  
**补充**：通过 **`system_bridge_init_netifs_from_hw` + `system_mode_manager_apply_saved_or_hw_default`**，在保持 IoT-Bridge **编译期能力集** 的前提下，对外设做 **运行时探测与默认模式落盘**，与上述结论一致。
